Data Collection, Processing & Exploration

📡

Data Collection Techniques

Methods, tools, and processes used to gather raw data from different sources — sensors, applications, logs, user interactions, devices, or external systems — for further storage, processing, and analysis. In cloud environments (like AWS), these techniques involve streaming, batch ingestion, transformation, and querying of data.

Key Characteristics

CharacteristicDescription
Source DiversityIoT devices, applications, databases, APIs, logs, or streams
Collection ModeReal-time (streaming) vs. batch (periodic uploads)
TransformationRaw data often needs to be filtered, cleaned, or structured
Storage DestinationS3, databases, or data warehouses
Analysis ReadinessPrepared for querying, dashboards, or machine learning
🌊

Amazon Kinesis

A fully managed AWS service for real-time data streaming and processing. Enables developers to ingest, analyze, and respond to large volumes of streaming data from IoT devices, application logs, social media feeds, and video streams.

Key Components

ComponentPurpose
Kinesis Data StreamsCaptures and processes real-time data streams at scale
Kinesis Data FirehoseDelivers streaming data to destinations like S3, Redshift, or Elasticsearch
Kinesis Data AnalyticsAnalyzes streaming data using SQL or Apache Flink in real time
Kinesis Video StreamsIngests and processes video streams for analytics and machine learning

How It Works

  1. Data Ingestion – Collects data from sensors, applications, or logs in formats like JSON or binary
  2. Sharding and Scaling – Splits incoming data into shards for parallel processing and fault tolerance; scales horizontally as needed
  3. Processing and Buffering – Applies transformations like filtering or aggregation before storing/forwarding
  4. Data Access and Delivery – Makes processed data available via APIs or delivers to other AWS services
λ

AWS Lambda

A serverless computing service that lets you run code in response to events without provisioning or managing servers.

Core Features

  • Event-driven execution – Automatically triggers functions in response to events from S3, DynamoDB, API Gateway, or Kinesis
  • Fully managed infrastructure – AWS handles server provisioning, scaling, patching, and monitoring
  • Automatic scaling – Functions scale up or down based on demand, even down to zero when idle
  • Flexible runtimes – Supports Node.js, Python, Java, Go, .NET, Ruby, and custom runtimes
  • MicroVM isolation – Uses Firecracker microVMs for secure, fast, and lightweight execution environments

How It Works

  1. Upload Code – Package code as a ZIP file or container image and upload to Lambda
  2. Define Triggers – Set up event sources like HTTP requests, file uploads, or database changes
  3. Configure Runtime & Permissions – Choose a runtime, allocate memory, and assign IAM roles
  4. Execute & Scale Automatically – Lambda runs function when triggered and scales based on traffic
🔍

AWS Athena

A serverless, interactive query service that lets you analyze data directly in Amazon S3 using standard SQL or Apache Spark.
  • Serverless architecture – No infrastructure to manage; pay only for queries run
  • SQL and Apache Spark support – Run ad-hoc queries using SQL or build analytics with Spark
  • Fast performance – Queries run in parallel and scale automatically for large datasets
  • Integrated with AWS Glue – Use Glue Data Catalog for metadata management and ETL operations
  • Notebook interface – Develop Spark applications using Python in a simplified notebook experience

Data Processing Paradigms

📦 Batch Processing

Collects data over a period of time, then processes it all at once periodically.

Used for: Payroll systems, data warehousing (ETL), end-of-day reporting, large-scale data transformation

Challenges: Not suitable for real-time; potential data staleness if intervals are long

🌊 Stream Processing

Continuously analyzes data as it arrives in real-time or near-real-time, in small chunks or individual records.

Used for: Fraud detection, monitoring systems, live dashboards, IoT applications

Challenges: Managing state, reliability, out-of-order data handling

📬

Message Queue

An asynchronous communication protocol where messages are stored in a queue until processed by a consumer. Decouples sender and receiver, allowing independent operation.
  • Microservices architectures for decoupling services
  • Task scheduling and background job processing
  • Event-driven systems
  • Ensuring reliable message delivery

Hadoop & MapReduce

Hadoop

An open-source framework for distributed storage and processing of large datasets using clusters of commodity hardware. Enables scalable, fault-tolerant data management.

MapReduce

A programming model and processing framework used to handle large-scale data across distributed clusters. Splits tasks into smaller units, processes them in parallel, and aggregates results efficiently. Based on Java.

Core Concepts

  • Map Task – Takes a set of data and converts it into key/value pair tuples (processes input data, breaks into small chunks)
  • Reduce Task – Takes map output, combines data tuples into a smaller set of tuples; always performed after map job
  • Shuffle Stage – Intermediate stage between Map and Reduce (sorting/grouping)
  • Mappers & Reducers – The data processing primitives under MapReduce model

Algorithm Stages

  1. Map Stage – Input data (from HDFS) passed to mapper function line by line; mapper processes and creates small data chunks
  2. Shuffle Stage – Sorts and groups mapped key/value pairs
  3. Reduce Stage – Reducer processes data from mapper; produces final output stored in HDFS
Key Advantage: Easy to scale data processing over multiple computing nodes. Scaling to hundreds or thousands of machines in a cluster is merely a configuration change — making it highly attractive.

Design Patterns / Applications

  • Index and Search
  • Classification
  • Recommendation (e.g., Sort)
  • Analytics (e.g., Join, Selection)
  • Summarization (e.g., Inverted Index)
  • Implemented by Google & Apache Hadoop
  • Works with HDFS, Pig, Hive, HBase
  • Large Scale Distributed Model

Apache Spark Streaming for IoT

A powerful extension of the Apache Spark framework enabling real-time data processing of streaming data. Handles live data streams from Kafka, Flume, Twitter, or TCP sockets using Spark's core APIs. Scalable, fault-tolerant, second-scale latencies.

How Spark Streaming Works

  1. Chops up data streams into batches of a few seconds
  2. Treats each batch as RDDs (Resilient Distributed Datasets) and processes using RDD operations
  3. Processed results are pushed out in batches

Key Concepts

  • Discretized Streams (DStreams) – Continuous data streams broken into micro-batches; each treated as an RDD
  • Input Sources – Apache Kafka, Flume, HDFS, Sockets, Custom data sources
  • Output Destinations – HDFS, Databases, Dashboards, External systems
  • RDD (Resilient Distributed Dataset) – Core data structure; fault-tolerant, distributed collection
📨

Apache Kafka

A publish-subscribe messaging system rethought as a distributed commit log. Characteristics: Fast, Scalable, Durable, Distributed.

Why Is Kafka So Fast?

  • Fast Writes – While Kafka persists all data to disk, essentially all writes go to the page cache
  • Fast Reads – Very efficient to transfer data from page cache to a network socket
  • Combined Effect – On a Kafka cluster where consumers are mostly caught up, there is no read activity on disks — serving data entirely from cache

Architecture Components

ComponentRole
ProducerPublishes messages to a Kafka topic
BrokerServer that stores and serves messages; multiple brokers form a Kafka cluster
ConsumerReads/subscribes to messages from topics
ZooKeeperManages and shares state among brokers, producers, and consumers
Partition / ShardTopics split into partitions for parallel processing; each partition has replicas

RabbitMQ

A lightweight, open-source message broker that facilitates communication between different parts of a system by passing messages through queues. Built on AMQP (Advanced Message Queuing Protocol). Uses FIFO (First In, First Out) message processing.

Core Concepts

ConceptDescription
ProducerSends messages to a queue
QueueStores messages until they are processed
ConsumerRetrieves and processes messages from the queue
ExchangeRoutes messages to queues based on rules (bindings). Types: Direct, Topic, Fanout
BindingConnects exchanges to queues with routing logic

Key Properties

  • Message-oriented middleware implementing AMQP
  • Developed and maintained by Apache Software Foundation
  • Supports multiple messaging protocols
  • Enables building distributed systems with asynchronous messaging
  • Widely used for asynchronous, decoupled, and reliable messaging in distributed apps

Predictive Modelling (PdM)

🎯

What is Predictive Modelling?

A technique that uses mathematical and computational methods to predict an event or outcome. Uses an equation-based model that describes the phenomenon, forecasting an outcome at some future state/time based on changes to model inputs. Model parameters explain how inputs influence outcomes.
Example: Predicting fuel efficiency based on a linear regression model of engine speed versus load.
📊

Data Requirements for PdM Modeling

Three main categories:

🎯

1. Relevancy

  • IoT deals with high volumes of data — challenging to identify which characteristics are actually relevant to the problem
  • For predicting motor failure, data must contain information related to motor operations
  • In large problem domains (like Predictive Maintenance), failures are often a composite of many components — better to design prediction systems around system components rather than the overall system
  • Important to leverage Domain Experts to identify which data subsets are most relevant
  • If no domain expert available, use Data Exploration and Feature Engineering techniques to identify relevant features
📏

2. Sufficiency

  • IoT provides high volumes of data, but under ML it's important to ensure the right type of data in the right level of abundance
  • Too much data → too high "data resolution" → difficult to process
  • Analogy: Image resolution — too low can't interpret, too high is costly to render
  • Machine life-spans are in the order of years — dataset must observe the system throughout the degradation process leading to failure
  • For rare/uncaptured failure events → use statistical analysis methods to cope with imbalanced dataset
✅

3. Quality

  • Ensure data is of high quality — "noise" must be minimized/eliminated
  • Use EDA (Exploratory Data Analysis) techniques to overcome Class Imbalance
  • If one class (e.g., failure events) represents less than 10% of the dataset → data is considered imbalanced
  • Various metrics can evaluate model effectiveness in case of class imbalance
⚖️

Handling Dataset Imbalance – Data Sampling Techniques

TechniqueApproachRisk/Drawback
Random Oversampling Replicate a random sample of minority class occurrences across the training dataset → increases minority class instances Overfitting — classifier becomes too specific, high training accuracy but poor performance on unseen test data
Random Under-sampling Remove majority class data to balance lower occurrences of minority class Classifier may miss important details → sub-optimal model performance
Hybrid Sampling Compromised approach — both oversample minority class AND undersample majority class simultaneously Balances both risks from above methods
💰

Cost-Sensitive Learning Techniques

Another method for handling data imbalance — incorporates "cost sensitivity" into machine learning algorithms. Assigns higher cost/weight to minority class examples during training.

📐 Support Vector Machines (SVM)

Assigns a cost of positive and negative examples during training. Comparable to a "weighted" pro-con list where negative examples hold more value than positive examples. Handles imbalanced data by weighting classes.

🌳 Boosted Decision Trees

An ensemble model leveraging gradient boosting to build a sequence of regression models where each model's output corrects the subsequent model. Leverages a composite of weaker prediction models to collectively improve overall prediction.

Feature Engineering over IoT Data

🛠️

What & When

"Feature engineering is creating a large and diverse set of derived variables from the base data." The richer the set of derived variables, the better the model. Quality of model depends on characteristics generation / variable generation.

Types of IoT Data

🔄 Dynamic Data (Streaming / Continuous)

Continuous stream with very small time intervals (minutes, seconds, milliseconds).
Examples: Temperature, pressure, voltage, current, flow rate, velocity, acceleration, heart rate, blood pressure

📋 Less Dynamic Data

Pertains to the overall state or working condition of a system.
Examples: Type of input materials being processed by a machine; treatment regime administered to a patient

🗃️

Aggregation of Data for Feature Engineering

  • Prior to creating features, consider the level of aggregation (across time) of the continuous streaming data
  • Atomic Level – Data used at the most granular level WITHOUT aggregation (e.g., every second)
  • Aggregated Level – Data aggregated over specific time windows to identify meaningful trends

⚡ Atomic Level (No Aggregation) — Example

Motor for a grinding machine: input particle size, temperature, flow rate data every second → predict optimal RPM within seconds/minutes.

If RPM too high → overheating → grinder damage
If RPM too low → decreased production rate

Used when: Very short prediction windows (process control)

📅 Aggregated Level — Example

Car driven for a few years: predicting component lifetime (days, weeks, months).
Sensors produce data frequently, but data is aggregated over time to understand meaningful trends signifying impending failure.

Used when: Longer prediction horizons; manufacturing equipment monitoring

Summary: Short prediction windows → atomic level. Longer prediction windows → aggregated level features prove more productive. Many cases use both.
🪟

Selecting the Optimal Time Window

Selecting the time window is often a critical consideration that drives the success of the feature engineering exercise in IoT. Four types:

Window TypeDescriptionUse Case / Note
Fixed Time Window Simplest window calculation. Aggregation over a specific, uniform time interval (e.g., 15 minutes across entire span) Uniform, consistent analysis
Variable Time Window Window size determined by a specific number of event occurrences (e.g., at least 4 instances of engine switch-on/off in vehicle telematics) Also helps treat missing values — define window as capturing at least N pings from sensor
Exponentially Expanding / Contracting Window size grows or shrinks exponentially over time Adaptive to signal behavior
Overlapping Window Try different types and lengths of windows; features across various windows used for building the model — "let the data pick what's most predictive" Most comprehensive; useful when optimal window is unknown
Missing Values: Variable time windows help treat missing values (e.g., sensor not transmitting due to satellite connectivity issues) by aggregating over sufficient data thresholds.
⚗️

Feature Generation Methods

Post aggregation, features are generated from the series (either aggregated or atomic). Five categories:

1️⃣

Basic Features — Change, Rate of Change, Standardization

If series values = X₁...Xₙ and time = t₁...tₙ:

Change: Cₘ₊₁ = (Xₘ₊₁ – Xₘ) / (tₘ₊₁ – tₘ)
Rate of Change: RTₘ = (Cₘ₊₁ – Cₘ) / (tₘ₊₁ – tₘ)
Growth / Decay: Gₘ₊₁ = (Xₘ₊₁ – Xₘ) / Xₘ
Rate of Growth/Decay: RGₘ = (Gₘ₊₁ – Gₘ) / (tₘ₊₁ – tₘ)
Moving Average: Average of (Xₘ₋ₚ to Xₘ)
Moving Std Dev: Std Dev of (Xₘ₋ₚ to Xₘ)
Relative Average: Moving Average / Global Average
Relative Std Dev: Moving Std Dev / Global Std Dev
  • Count of values above or below a threshold value
  • Ratio of changes, growth rate etc. with standard deviation
  • Features involving trend of values across various aggregation windows
2️⃣

Features Based on Relationships

  • Leverages data from multiple sensors measuring similar or dissimilar quantities
  • Natural relationships may exist between multiple signals; deviations from expected relationship may indicate operating condition changes or breakdowns
  • Method: Build a regression model (e.g., temperature as Y; RPM, particle size, flow rate as Xs) → compute error series → under "un-natural" conditions, error or its std dev shoots up
  • Grinder Example: 16 thermocouples + RPM + flow rates + particle size; temperature expected as a function of RPM and flow rates — deviations from expected indicate hard impurities in material
3️⃣

Features Based on Higher Order Statistics (Moments)

  • Usually moments (mean, variance, skewness, kurtosis) calculated within the aggregation window
  • For non-Gaussian series → use cumulants rather than moments for information about the distribution
  • Cumulants defined using the cumulant-generating function = natural logarithm of the moment-generating function
  • Till 3rd cumulant: value of moment = cumulant; from 4th onwards, cumulants ≠ moments
4️⃣

Features Based on Outlier Detection / Anomaly Detection / Fault Detection

  • Detect outliers and use their presence within an aggregation window as features
  • Kalman Filter – Most popular method for outlier detection
  • PCA Method – Create principal component analysis using existing features; identify distance of each point from origin of multi-dimensional space; points ≥ 3 std deviations from origin → outlier
  • Cook Distance – Used in regression-type approach for outlier detection
5️⃣

Features Based on Series Transformation

  • Continuous streaming data treated as a signal in the time domain
  • Mathematical transformations decompose this signal into simpler functions → analyzed across windows to identify changes in pattern

Common Transformations

TransformDescription
Fast Fourier Transform (FFT)Transforms signal from time domain → frequency domain. Decomposes into sinusoidal functions of the form A·Cos(ωt) + Bj·Sin(ωt). Calculates amplitude, phase, and frequencies. Intuitively: like breaking music into component notes. Most commonly used.
Hilbert Huang TransformAdaptive method for non-linear/non-stationary signals
Wigner Ville DistributionTime-frequency representation
Wavelet TransformationMulti-resolution analysis of signals
FFT for Feature Engineering: Compute FFT for each aggregation window → get frequencies and amplitudes of component sinusoids → use amplitude series for further feature generation. A new frequency in any window may indicate anomalous behavior.
